# jarvis 个人 Agent 规划

## 1. 产品定位

`jarvis` 是我的综合个人助理，也是我的个人 Agent OS。

它的目标不是只做一个聊天机器人，而是作为我的统一智能中枢，连接我的生活管理、投资研究、通用情报获取、AI 研究、知识库整理、编码执行和自媒体创作，并通过可扩展的工具系统、长期记忆系统、工作流编排能力和 GUI 管理后台，形成一个可持续维护和进化的个人智能系统。

---

## 2. 核心目标

jarvis 需要帮助我完成以下目标：

- 统一处理生活、工作、学习、投资和创作类任务
- 能够主动调用工具，而不只是被动回答问题
- 能够持续积累我的偏好、上下文和长期目标
- 能够把外部信息转化为可复用的知识资产
- 能够辅助开发、生成和维护我自己的工具
- 能够通过 GUI 对所有能力进行可视化管理和维护

---

## 3. jarvis 的角色定义

jarvis 可以被定义为三层结构：

### 3.1 jarvis Core

作为总控中枢，负责：

- 统一身份与用户画像
- 长期记忆和偏好管理
- 任务理解与路由分发
- 工作流调度
- 工具调用与结果整合
- 权限控制与风险控制
- 执行日志与行为审计

### 3.2 Domain Agents

按照不同领域拆分子能力，而不是让一个 Agent 负责所有事情。

建议包含以下领域代理：

- 生活助理 Agent
- 投资助理 Agent
- 通用情报 Agent
- AI 研究 Agent
- 知识库助理 Agent
- 编码助理 Agent
- 自媒体创作 Agent

建议额外预留一个扩展位：

- 关系与协作 Agent（按需要启用）

其中，`AI 研究 Agent` 是 jarvis 的前沿情报与方法研究单元，负责持续跟踪 AI 领域的模型、论文、产品、工具链、开源项目和关键创作者，将外部变化转化为对我有用的趋势判断、实验方向、工具机会、内容选题和投资研究输入；它不负责通用信息采集、最终归档、代码实现或发布执行，而是专注于 AI 主题的深度研究、方法沉淀和战略判断。

`关系与协作 Agent` 作为预留扩展模块，负责管理联系人、沟通上下文、会议后的跟进事项、合作机会和外部关系网络，将“人脉与协作”沉淀为长期资产；它不负责生活事务管理、投资判断、代码实现或未经确认的外部敏感发送，只有当 jarvis 开始承担较多会议、合作、邮件、社交关系维护任务时再正式启用。

### 3.2.1 Agent 分层

为了避免所有 Agent 平铺后互相重叠，建议把 jarvis 的领域 Agent 再分成三层：

- `事务执行层`：生活助理 Agent、编码助理 Agent、自媒体创作 Agent
- `情报研究层`：通用情报 Agent、AI 研究 Agent、投资助理 Agent
- `资产沉淀层`：知识库助理 Agent
- `关系协作扩展层`：关系与协作 Agent（按需启用）

这三层分别回答三个问题：

- 事务执行层：现在要做什么
- 情报研究层：外部发生了什么、为什么重要
- 资产沉淀层：哪些内容值得长期保存和复用

这样设计的好处是：

- 把“采集”“判断”“执行”“归档”拆开
- 避免一个 Agent 既研究又执行又归档
- 方便后续扩展新的专业 Agent

### 3.2.2 Agent 角色定位

建议对 7 个核心 Agent 以及 1 个预留扩展 Agent 的角色做进一步收敛：

| Agent | 角色定位 | 主要产出 | 不负责 |
|---|---|---|---|
| 生活助理 Agent | 日常运营者 | 日程、待办、提醒、生活记录 | 深度研究、代码实现 |
| 投资助理 Agent | 投资研究协调者 | 标的跟踪、风险提示、投资复盘 | 通用信息抓取、自动交易 |
| 通用情报 Agent | 外部信息扫描器 | 新闻摘要、专题情报包、订阅流整理 | AI 专题深研、最终归档 |
| AI 研究 Agent | AI 专题研究员 | 趋势判断、方法论、实验方向 | 全网泛采集、最终发布 |
| 知识库助理 Agent | 系统馆长 | 分类、标签、实体、关系、归档 | 原始研究判断、内容发布 |
| 编码助理 Agent | 工具建造者 | 脚本、工具、代码改动、自动化实现 | 内容策略、知识最终归档 |
| 自媒体创作 Agent | 内容生产者 | 选题、提纲、草稿、多平台改写 | 长期知识治理、代码实现 |
| 关系与协作 Agent | 外部关系协调者 | 联系人档案、沟通记录、合作跟进、关系网络 | 日常生活管理、投资判断、直接敏感外发 |

### 3.2.3 关系与协作 Agent 的启用条件

`关系与协作 Agent` 不必在 jarvis 初期就独立运行，但建议在以下场景满足其中三项及以上时正式启用：

- 高频会议和多方协作明显增加
- 需要持续维护联系人档案和沟通记录
- 需要跟踪合作机会、商务往来或外部资源
- 内容创作开始涉及品牌、投稿、采访或合作对象
- 投资或项目推进开始依赖行业联系人和关系网络
- 经常出现“会后没有系统跟进”或“联系人信息散落”问题

启用后的协作位置建议为：

- 上游接收生活助理 Agent 的会议与提醒输入
- 横向连接投资助理 Agent、自媒体创作 Agent 和知识库助理 Agent
- 下游输出联系人资产、协作上下文和待跟进事项

### 3.3 Tool & Workflow Layer

作为 jarvis 的执行层，负责：

- 调用各种工具
- 编排固定流程
- 执行自动化任务
- 管理输出结果
- 记录执行历史

## 4. 七大核心能力域

### 4.1 生活管理

目标：让 jarvis 成为日常生活运营助手。

包含能力：

- 日程整合
- 待办管理
- 提醒与周期性复盘
- 消费记录与账单理解
- 出行、会议、行程安排
- 生活信息查询与归档

### 4.2 投资管理

目标：让 jarvis 成为投资研究与风险提示助手。

包含能力：

- 市场资讯聚合
- 自选标的跟踪
- 投资观点归档
- 财经内容摘要
- 风险事件提醒
- 周期性复盘报告生成

原则：

- jarvis 提供信息整合、研究辅助和风险提示
- 高敏感交易动作必须设置人工确认

### 4.3 通用情报获取

目标：把外部海量信息转化为有结构、可筛选、可分发的通用情报。

包含能力：

- 新闻、博客、社媒、论坛信息抓取
- 信息订阅源管理
- 自动摘要与去重
- 主题分类与优先级排序
- 周报、日报、专题包生成
- 自动写入知识库

### 4.4 AI 研究

目标：让 jarvis 成为我的 AI 前沿研究员和方法论追踪助手。

包含能力：

- 跟踪 AI 模型、论文、产品、开源项目和行业动态
- 识别值得长期关注的技术主题、方法路线和产品趋势
- 输出 AI 周报、专题研究和趋势判断
- 将 AI 外部变化转化为工具机会、自动化方向和实验课题
- 为投资研究、内容创作和开发决策提供 AI 主题输入
- 沉淀对我有价值的 AI 方法论、工作流和最佳实践

边界：

- 不负责广义的全网信息采集
- 不替代知识库 Agent 做最终归档和整理
- 不替代编码 Agent 做具体实现
- 不替代内容 Agent 做最终发布

### 4.5 知识库整理

目标：把输入的信息沉淀成长期资产。

包含能力：

- 文档收集与清洗
- 摘要与标签生成
- 实体识别与主题聚合
- 双向链接与关联推荐
- 项目资料归档
- 定期复习与更新提醒

知识需要分为两类：

- `事实记忆`：个人资料、偏好、项目背景、联系人、投资标的、长期规则
- `工作记忆`：当前上下文、临时任务、草稿、近期对话和执行状态

### 4.6 编码与工具开发

目标：让 jarvis 不只是使用工具，也能帮助我持续构建工具。

包含能力：

- 编写和修改代码
- 自动生成脚本和小工具
- 调试、测试与排错
- API 接入与数据处理
- 自动化任务开发
- 工具版本管理和配置维护

### 4.7 自媒体创作

目标：让 jarvis 形成内容生产闭环。

包含能力：

- 选题发现
- 素材搜集
- 资料摘要
- 提纲生成
- 初稿写作
- 多平台改写
- 发布排期
- 数据复盘

## 5. jarvis 的系统原则

为了确保 jarvis 可长期使用，系统设计需要遵循以下原则：

### 5.1 统一入口

所有任务都尽量通过 jarvis 进入，避免能力分散在多个脚本、页面或 Bot 中。

### 5.2 领域拆分

jarvis 是中控，不是所有事情都自己做。不同领域能力应该有清晰职责边界。

### 5.3 可观测

每次执行都应该可以看到：

- 谁执行了任务
- 调用了哪些工具
- 输入是什么
- 输出是什么
- 是否失败
- 是否需要人工确认

### 5.4 可维护

Prompt、工具、工作流、知识库和记忆，都必须能通过 GUI 配置和维护。

### 5.5 可进化

jarvis 应该具备不断增加工具、扩展流程、优化规则和积累经验的能力。

### 5.6 单任务单主责

每一个任务在任意时刻都应有一个“主责 Agent”。

其他 Agent 可以提供输入，但不应在同一任务上与主责 Agent 争夺控制权。

例如：

- “写一篇 AI 工具分析文章”的主责 Agent 是自媒体创作 Agent
- AI 研究 Agent 提供趋势判断和方法观点
- 通用情报 Agent 提供素材
- 知识库 Agent 负责最终归档

### 5.7 研究与执行分离

建议明确区分：

- `研究型 Agent`：负责发现、分析、判断、提出建议
- `执行型 Agent`：负责产出、修改、调用工具、完成动作

这条原则能避免：

- 研究结论直接变成高风险执行
- 情报 Agent 越权修改核心资料
- 内容 Agent 越权做投资判断

### 5.8 归档由知识库 Agent 收口

无论资料来自哪个 Agent，最终进入长期资产层时，都应由知识库 Agent 或知识库流程统一收口。

这样做的目的：

- 保持标签和实体结构统一
- 保持知识目录整洁
- 避免每个 Agent 各自定义归档标准

### 5.9 默认协作链

建议 jarvis 中的大多数复杂任务遵循以下基础链路：

`通用情报 Agent / AI 研究 Agent / 投资助理 Agent`  
-> `知识库助理 Agent`  
-> `编码助理 Agent 或 自媒体创作 Agent 或 生活助理 Agent`

可以理解为：

- 先发现和判断
- 再沉淀为资产
- 再进入执行或输出

### 5.10 高风险动作必须最后一跳确认

当任务涉及以下行为时，不应由研究型 Agent 直接触发：

- 发布
- 删除
- 外部发送
- 交易
- 改写关键配置

这些动作必须由执行型 Agent 发起，并经过人工确认或明确权限控制。

### 5.11 对外沟通与关系动作单独治理

如果启用 `关系与协作 Agent`，建议将以下动作视为单独治理范围：

- 发邮件
- 发私信
- 对外邀约
- 合作跟进
- 发送联系人摘要

原因是这类动作虽然不一定涉及金钱，但会直接影响外部关系，因此应具备：

- 明确审批规则
- 清晰发送前预览
- 完整沟通日志
- 联系人与项目的关联记录

### 5.12 典型任务路由

为了让 jarvis 的多 Agent 协作具备稳定运行逻辑，建议为高频任务定义默认路由。

#### 路由字段说明

- `主责 Agent`：对任务结果负责的 Agent
- `协作 Agent`：提供素材、研究、上下文或辅助执行的 Agent
- `归档 Agent`：负责将结果纳入长期资产层的 Agent
- `审批点`：需要人工确认或敏感治理的节点

#### 高频任务默认路由表

| 任务类型 | 主责 Agent | 协作 Agent | 归档 Agent | 审批点 |
|---|---|---|---|---|
| 日程整理与提醒 | 生活助理 Agent | 无或关系与协作 Agent | 知识库助理 Agent（可选） | 一般无 |
| 会议纪要与后续跟进 | 生活助理 Agent | 关系与协作 Agent、知识库助理 Agent | 知识库助理 Agent | 对外跟进前确认 |
| 每日资讯摘要 | 通用情报 Agent | AI 研究 Agent | 知识库助理 Agent | 一般无 |
| AI 周报 / AI 趋势研究 | AI 研究 Agent | 通用情报 Agent、自媒体创作 Agent | 知识库助理 Agent | 对外发布前确认 |
| 投资标的复盘 | 投资助理 Agent | 通用情报 Agent、AI 研究 Agent | 知识库助理 Agent | 涉及交易时确认 |
| 生成研究笔记 | AI 研究 Agent 或 投资助理 Agent | 通用情报 Agent | 知识库助理 Agent | 一般无 |
| 知识整理与标签统一 | 知识库助理 Agent | 所有上游 Agent | 知识库助理 Agent | 一般无 |
| 开发工具或脚本 | 编码助理 Agent | AI 研究 Agent、知识库助理 Agent | 知识库助理 Agent | 改关键配置时确认 |
| 内容选题挖掘 | 自媒体创作 Agent | 通用情报 Agent、AI 研究 Agent | 知识库助理 Agent | 一般无 |
| 内容初稿写作 | 自媒体创作 Agent | 通用情报 Agent、AI 研究 Agent | 知识库助理 Agent | 发布前确认 |
| 多平台改写与发布准备 | 自媒体创作 Agent | 知识库助理 Agent | 知识库助理 Agent | 发布前确认 |
| 联系人维护与合作跟进 | 关系与协作 Agent | 生活助理 Agent、知识库助理 Agent | 知识库助理 Agent | 对外发送前确认 |

#### 关键路由模式

可以进一步把任务路由抽象成几种常见模式：

##### 模式 A：情报 -> 研究 -> 归档

适用于：

- AI 主题跟踪
- 行业专题研究
- 投资线索整理

默认链路：

`通用情报 Agent` -> `AI 研究 Agent / 投资助理 Agent` -> `知识库助理 Agent`

##### 模式 B：情报 / 研究 -> 创作 -> 归档

适用于：

- 文章写作
- 视频脚本
- 社媒内容策划

默认链路：

`通用情报 Agent / AI 研究 Agent` -> `自媒体创作 Agent` -> `知识库助理 Agent`

##### 模式 C：研究 -> 开发 -> 归档

适用于：

- 新工具开发
- 自动化脚本实现
- 基于新 AI 方法的实验

默认链路：

`AI 研究 Agent / 知识库助理 Agent` -> `编码助理 Agent` -> `知识库助理 Agent`

##### 模式 D：会议 / 关系 -> 跟进 -> 归档

适用于：

- 会议后行动项
- 联系人维护
- 合作推进

默认链路：

`生活助理 Agent / 关系与协作 Agent` -> `关系与协作 Agent` -> `知识库助理 Agent`

#### 路由冲突处理原则

如果多个 Agent 都“看起来能做”，优先按以下规则判定主责：

1. 谁最接近最终输出物，谁主责。
2. 谁承担高风险动作，谁必须成为最后一跳。
3. 涉及长期资产入库时，知识库助理 Agent 负责收口。
4. 涉及对外沟通时，关系与协作 Agent 或自媒体创作 Agent 负责最后输出，不由研究型 Agent 直接外发。

#### 当前推荐的默认工作方式

jarvis 当前最稳的运行方式不是“所有 Agent 随时自由协作”，而是：

- 先确定一个主责 Agent
- 再决定需要哪些协作 Agent
- 最后由知识库助理 Agent 统一归档

这样可以保证：

- 路由稳定
- 责任清晰
- 资料不散
- 风险可控

## 6. GUI 管理后台规划

GUI 不是单纯的聊天界面，而应该是 jarvis 的控制台。

建议最少包含以下模块：

### 6.1 Dashboard 总览

用于展示整体状态：

- 今日任务
- 今日提醒
- 最近执行记录
- 重要风险提示
- 新获取的信息摘要
- 待审批动作
- 当前各 Agent 状态

### 6.2 Agents 管理

用于维护领域 Agent：

- Agent 名称
- 职责定义
- 系统提示词
- 可调用工具范围
- 默认工作流
- 权限等级
- 最近运行记录

### 6.3 Tools 管理

这是整个系统非常关键的模块。

每个工具建议具备以下字段：

- 工具名称
- 工具类型
- 功能描述
- 输入参数 schema
- 输出结果 schema
- 依赖服务或 API
- 权限级别
- 是否启用
- 版本号
- 调用次数
- 最近错误日志

工具建议按三类管理：

- `系统工具`：文件、Shell、浏览器、数据库、通知
- `业务工具`：投资分析、知识整理、内容发布、数据抓取
- `实验工具`：临时脚本、小型自动化、测试型能力

### 6.4 Workflows 管理

用于管理自动化流程和固定任务模板。

建议支持：

- 创建流程模板
- 步骤编排
- 条件分支
- 定时触发
- 人工审批节点
- 失败重试
- 执行历史查看

示例流程：

- 每日信息抓取 -> 摘要 -> 去重 -> 分类 -> 写入知识库 -> 生成日报
- 每周投资复盘 -> 汇总新闻 -> 对照持仓 -> 生成复盘报告
- 创作工作流 -> 选题 -> 素材收集 -> 提纲 -> 初稿 -> 多平台改写

### 6.5 Knowledge Base 管理

用于可视化维护知识库。

建议功能：

- 文档列表
- 标签体系
- 分类体系
- 来源追踪
- 更新时间线
- 主题聚合
- 实体关系
- 手动修订入口

### 6.6 Memory 管理

用于维护 jarvis 对“我”的理解。

建议管理内容：

- 个人画像
- 偏好设定
- 长期目标
- 项目背景
- 联系人档案
- 风险偏好
- 禁止事项和边界条件

### 6.7 Content Center 内容中心

用于管理自媒体与内容资产。

建议功能：

- 选题池
- 素材池
- 草稿箱
- 已发布内容
- 平台版本管理
- 数据表现分析

### 6.8 Audit & Permission 审计与权限

这是系统可信度的核心。

建议功能：

- 工具调用日志
- 敏感操作日志
- 审批记录
- 错误日志
- 回滚记录
- 凭证管理
- 权限分级配置

补充说明：

- 通用管理后台负责系统总控
- Workspace GUI 单独承担资料浏览、文件组织、知识可视化和迁移管理
- Workspace GUI 详细设计建议独立成文档维护

## 7. 工具管理体系

由于 jarvis 会不断新增工具，因此需要一个清晰的工具管理机制。

### 7.1 工具注册

每个工具都应该有统一注册信息：

- `id`
- `name`
- `description`
- `category`
- `input_schema`
- `output_schema`
- `permissions`
- `version`
- `owner`
- `status`

### 7.2 工具生命周期

建议工具生命周期如下：

1. 设计
2. 开发
3. 测试
4. 注册
5. 启用
6. 监控
7. 迭代或下线

### 7.3 工具治理

需要支持：

- 工具启停
- 版本切换
- 权限限制
- 调用统计
- 错误追踪
- 文档说明
- 依赖管理

### 7.4 工具与 Agent 的关系

不是所有 Agent 都能调用所有工具。

应采用：

- Agent 与工具的白名单机制
- 按领域授权
- 按敏感级别限制
- 按流程节点限制

## 8. 操作权限分级

为了防止 jarvis 在高风险场景失控，建议引入四级操作模型。

### L1 查询

只读操作，不写入，不外发。

例如：

- 查资料
- 搜索信息
- 阅读知识库
- 查看日志

### L2 整理

允许生成和整理，但不直接执行外部高风险动作。

例如：

- 摘要内容
- 归档知识
- 生成草稿
- 生成报告

### L3 执行

允许进行内部执行动作。

例如：

- 创建任务
- 更新知识库
- 运行脚本
- 修改配置

### L4 敏感操作

涉及金钱、发布、删除、对外发送、账号操作，必须人工确认。

例如：

- 发布内容
- 发邮件或消息
- 删除文件
- 触发交易
- 修改关键权限

## 9. 建议的数据与系统模块

从系统实现上，可以拆成以下核心模块：

- `workspace service`：个人工作区、目录规则、文件落盘与索引
- `profile service`：用户画像与偏好管理
- `memory service`：长期记忆与工作记忆
- `agent router`：任务理解与路由
- `tool registry`：工具注册与发现
- `workflow engine`：流程编排与自动化执行
- `knowledge service`：知识入库、检索、关联
- `content service`：内容创作资产管理
- `audit service`：日志、权限与审计
- `gui console`：统一可视化管理入口

## 10. Personal Workspace 与资料管理

jarvis 不应该只是一组 Agent，还需要建立在统一的个人工作区之上。

这个工作区的意义是：

- 让所有 Agent 产出的资料都有明确落点
- 让文件可以被 VSCode、Obsidian、Finder 直接读取
- 让资料能够备份、迁移、同步和长期保管
- 让 GUI 管理的是同一批真实文件，而不是分散在多个黑盒系统中的数据

### 10.1 核心原则

#### 文件系统是真相源

所有重要资料都应优先落地为普通文件，而不是只存在于数据库或向量索引中。

建议优先格式：

- `md`
- `json`
- `yaml`
- `csv`
- `pdf`
- `png`
- 代码文件

#### 内容与视图分离

jarvis GUI、VSCode、Obsidian 都只是工作区的不同视图，不应成为资料的唯一入口。

#### Agent 可变，资料结构应稳定

文件存储应尽量按照领域、资料类型和状态组织，而不是完全按 Agent 名称组织。

#### 可迁移优先

任何时候都应可以通过复制整个工作区目录完成迁移，而不依赖某个特定前端或服务。

### 10.2 建议的工作区结构

建议采用单一根目录管理 jarvis 资料，例如：

```text
jarvis/
  00-system/
    config/
    registry/
    prompts/
    workflows/
    logs/
    backups/

  01-inbox/
    capture/
    imports/
    unsorted/

  10-life/
    calendar/
    tasks/
    journal/
    finance-personal/

  20-invest/
    watchlist/
    research/
    market-notes/
    thesis/
    reviews/

  30-intel/
    news/
    web-clips/
    digests/
    topics/

  40-knowledge/
    notes/
    projects/
    entities/
    references/
    maps/

  50-dev/
    projects/
    scripts/
    tools/
    specs/

  60-content/
    ideas/
    outlines/
    drafts/
    published/
    assets/

  90-archive/
    snapshots/
    exports/
    deprecated/
```

### 10.3 资料落盘规则

不同 Agent 产出的内容，应落到稳定的领域目录中：

- 投资 Agent 的研究资料放入 `20-invest/research/`
- 信息 Agent 的新闻摘要放入 `30-intel/digests/`
- 知识整理 Agent 的主题笔记放入 `40-knowledge/notes/`
- 编码 Agent 的脚本和工具放入 `50-dev/scripts/` 或 `50-dev/tools/`
- 创作 Agent 的提纲和草稿放入 `60-content/outlines/` 与 `60-content/drafts/`

不建议把所有内容放入类似 `agents/invest-agent/outputs/` 这类目录，因为 Agent 本身可能调整，而资料归属通常更稳定。

### 10.4 命名与 Metadata 规范

每份核心资料都建议具备统一 metadata，以支持可检索、可迁移和可视化。

例如 Markdown frontmatter：

```md
---
id: invest-note-2026-04-13-001
title: 英伟达供应链观察
created_at: 2026-04-13
updated_at: 2026-04-13
agent: invest-agent
domain: invest
type: research-note
status: draft
source_type: web
source_url: https://example.com
tags: [invest, semis, nvidia]
visibility: private
---
```

建议至少统一以下字段：

- `id`
- `title`
- `created_at`
- `updated_at`
- `agent`
- `domain`
- `type`
- `status`
- `tags`
- `source_type`
- `source_url`

### 10.5 索引层设计

虽然文件系统是长期真相源，但仍然需要索引层来支持 GUI 与智能检索。

索引层建议负责：

- 解析文件 metadata
- 建立全文索引
- 建立实体与关系索引
- 记录文件与 Agent、Workflow、Tool 的关联
- 记录最近变更和版本历史

建议原则：

- 索引可以重建
- 原始文件不可丢
- 索引不替代原始资料

### 10.6 与 VSCode、Obsidian 的关系

建议明确三者定位：

- `VSCode`：代码、脚本、配置、批量编辑
- `Obsidian`：知识浏览、链接关系、主题沉淀、人工整理
- `jarvis GUI`：系统管理、资料总览、工作流控制、可视化检索、迁移管理

三者面向同一工作区，不应该各自持有独立孤岛数据。

### 10.7 备份与迁移策略

为了满足长期保管和跨设备迁移，建议从一开始就支持：

- 根目录级备份
- 文本资料版本管理
- 定期 snapshot 导出
- 工作区 manifest 文件
- 相对路径资源引用

建议最少具备：

- `00-system/backups/`：定期备份
- `00-system/registry/`：记录 Agent、Tool、Workflow 定义
- `00-system/config/`：记录系统配置
- `90-archive/snapshots/`：按时间快照导出

### 10.8 Workspace GUI 的职责

Workspace GUI 应专注于“工作区与资料层”，而不是重复承担整个系统后台的职责。

它的重点包括：

- 文件和资料浏览
- 按领域、时间、Agent、标签筛选
- 文档与资产预览
- 知识图谱与时间线
- 备份、迁移和同步状态展示

Workspace GUI 详细设计建议见独立文档。

## 11. MVP 建议

不要一开始就做成完整平台，可以先做第一阶段最小可用版本。

### Phase 1：基础版

目标：先把 jarvis 跑起来。

包含：

- 统一聊天入口
- 基础长期记忆
- 生活、通用情报、知识、编码 4 个基础 Agent
- 工具注册页
- 基础工作流页
- 执行日志页
- 基础工作区目录和 metadata 规范

### Phase 2：增强版

目标：把系统从可用变成可管理。

包含：

- 完整 Dashboard
- 知识库可视化
- 内容中心
- 投资模块
- 权限审批机制
- 工具版本与错误监控
- Workspace GUI
- 文件索引和知识关系视图

### Phase 3：成熟版

目标：把 jarvis 变成个人 Agent OS。

包含：

- 更细粒度的多 Agent 协作
- 更稳定的自动化工作流
- 主动提醒和主动任务建议
- 内容生产闭环
- 跨设备统一控制台
- 更强的个性化与长期学习能力
- 完整迁移、备份和恢复能力

## 12. 一句话定义

`jarvis` 是以我为中心的个人 Agent OS，通过统一中枢、领域代理、工具系统、工作流引擎、长期记忆和 GUI 控制台，把生活、投资、信息、知识、开发与创作整合进一个可持续维护和进化的智能系统。

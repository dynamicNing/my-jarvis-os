# jarvis Workspace GUI 规划

## 1. 定位

`Workspace GUI` 是 jarvis 的资料工作台，负责把个人工作区中的文件、知识、资产和执行痕迹可视化。

它不是单纯的聊天界面，也不是通用后台管理页，而是面向“资料保管、迁移、检索、浏览、整理”的专用界面。

它应服务于以下目标：

- 让我直观看到 jarvis 产生和管理的所有资料
- 让我用图形界面管理文件、知识、标签、关系和状态
- 让我确认这些资料真实保存在本地工作区中
- 让我可以随时备份、导出和迁移整个系统

---

## 2. 产品目标

Workspace GUI 需要解决的核心问题：

- 资料散落在不同目录中，不易统一浏览
- Agent 产出的内容难以追踪来源与归属
- Obsidian 和 VSCode 虽可用，但缺少全局管理视角
- 文件虽然存在本地，但缺少面向知识和任务的可视化方式
- 缺少针对迁移、备份、快照和同步状态的统一界面

---

## 3. 设计原则

### 3.1 文件优先

GUI 管理的是工作区中的真实文件，而不是只存在于数据库中的副本。

### 3.2 多视图，不改真相源

列表、看板、图谱、时间线都只是不同视图，底层仍然是同一个工作区。

### 3.3 先可读，再智能

即使不依赖 AI，Workspace GUI 也应该让用户直接看懂资料结构、文件状态和上下游关系。

### 3.4 可迁移优先

任何可视化都不应依赖专有格式，保证切换设备或切换系统时不会损失核心资产。

---

## 4. 第一阶段边界

`Workspace GUI` 的第一阶段定位为只读工作台。

这意味着 Phase 1 只负责：

- 浏览工作区内容
- 展示索引结果
- 预览文档与资产
- 提供筛选、跳转和可视化视图

这意味着 Phase 1 不负责：

- 直接修改 metadata
- 移动或重命名文件
- 批量归档
- 删除文件
- 执行任何会改写工作区的操作

所有写操作都放到后续阶段再定义。

---

## 5. 管理对象

Workspace GUI 主要管理以下对象：

- 文件
- 文件夹
- 笔记
- 媒体资产
- 项目资料
- Agent 产出物
- Workflow 执行结果
- Tags
- Entities
- Collections
- Snapshots

---

## 6. 页面结构

建议包含以下核心页面：

### 6.1 Workspace Home

作为工作区首页，展示：

- 最近新增文件
- 最近更新资料
- 待整理资料
- 最近运行的 Agent 产出
- 最近执行的 Workflow 输出
- 快照与备份状态
- 存储空间概览

### 6.2 Explorer

这是最核心的页面，用来替代单纯的文件树浏览。

应支持：

- 目录树浏览
- 文件列表视图
- 卡片视图
- 按领域筛选
- 按 Agent 筛选
- 按时间筛选
- 按标签筛选
- 按状态筛选
- 快速预览

### 6.3 Document Viewer

用于查看单个资料对象。

应支持：

- Markdown 预览
- Frontmatter 展示
- 来源信息展示
- 相关文件推荐
- 引用关系
- Agent 生成记录
- 历史版本入口（预留）

### 6.4 Knowledge Graph

用于可视化知识与资料关系。

节点可以包括：

- 文档
- 人物
- 项目
- 公司
- 投资标的
- 主题
- 内容选题

关系可以包括：

- 引用
- 属于
- 提及
- 来源于
- 派生自
- 关联到

### 6.5 Timeline

用于按时间回看工作区活动。

应展示：

- 哪个 Agent 在什么时间产出什么内容
- 哪个 Workflow 修改了哪些文件
- 哪些资料最近被归档、更新或发布
- 哪些快照被创建

### 6.6 Collections

用于把跨目录资料按主题重新组织。

例如：

- AI 主题
- 某个投资标的
- 某个长期项目
- 某个创作系列

它本质上是虚拟聚合，不改变底层真实目录，也不在 Phase 1 写回文件系统。

### 6.7 Assets

用于查看图片、PDF、网页截图、音视频等非文本资料。

应支持：

- 缩略图浏览
- 按类型筛选
- 来源定位
- 关联文档查看
- 批量归档

### 6.8 Backups & Migration

这是 Workspace GUI 的差异化重点页面。

应支持：

- 最近快照列表
- 备份时间线
- 导出包记录
- 工作区完整性检查
- 路径健康检查
- 缺失资源提醒
- 恢复点管理（后续阶段细化）

---

## 7. 核心视图模式

同一批资料建议支持以下几种视图：

- `List View`：适合筛选、排序、批量管理
- `Board View`：适合按状态或流程阶段管理
- `Graph View`：适合查看关系网络
- `Timeline View`：适合查看时间演化
- `Split View`：左侧目录，右侧内容预览

---

## 8. 数据筛选维度

Workspace GUI 应支持统一筛选体系：

- 领域
- Agent
- 文件类型
- 标签
- 状态
- 时间范围
- 来源
- Workflow
- 是否已归档
- 是否需要人工整理

---

## 9. 索引与同步机制（简化版）

`Workspace GUI` 不直接把数据库当真相源，而是依赖独立索引服务读取工作区文件并生成可查询结果。

简化方案建议如下：

- 原始文件系统始终是真相源
- 启动时执行一次全量扫描
- 之后按固定周期执行定时扫描
- GUI 默认读取索引结果，而不是直接拼装原始文件
- 当索引未就绪时，界面应明确展示状态，而不是静默显示空白

第一阶段只使用最简单的索引状态：

- `未扫描`
- `扫描中`
- `可用`
- `扫描异常`

这已经足够支撑 MVP，不必在当前版本引入复杂事件流、文件监听或冲突合并机制。

数据契约暂时保留为待定项，在索引服务真正落地前单独定义。

---

## 10. 单文件详情结构

单个文件的详情页建议包含以下区域：

- 基本信息
- 文件路径
- 创建时间和更新时间
- Agent 来源
- Workflow 来源
- 标签
- 状态
- 内容预览
- 关联对象
- 历史版本入口（预留）
- 操作记录

---

## 11. 与 Obsidian 和 VSCode 的关系

Workspace GUI 不替代它们，而是提供更高层的工作区视角。

建议关系如下：

- `Obsidian`：查看和整理知识笔记
- `VSCode`：编辑代码、脚本、配置和原始文件
- `Workspace GUI`：总览、筛选、关系视图、迁移备份、资产管理

可增加快捷动作：

- 在 VSCode 中打开
- 在 Obsidian 中打开
- 打开所在目录
- 查看原始文件

---

## 12. 迁移与备份可视化

你特别关心“保管、迁移、可视化”，所以这部分应该是 Workspace GUI 的关键能力。

建议可视化展示：

- 当前工作区根目录
- 最近一次完整备份时间
- 最近一次快照时间
- 可恢复的历史节点
- 文件总量和增长趋势
- 大文件占用情况
- 缺失 metadata 文件比例
- 无归类文件数量

还可以增加：

- 一键导出 manifest
- 一键生成迁移包
- 迁移前一致性检查
- 路径修复建议

---

## 13. Workspace GUI 与系统后台的边界

建议明确区分：

`系统后台` 负责：

- Agent 管理
- Tool 管理
- Workflow 管理
- 权限管理
- 审计管理

`Workspace GUI` 负责：

- 文件工作区浏览
- 资料可视化
- 知识关系图
- 时间线
- 资产管理
- 备份与迁移视图

这样能避免一个后台把所有功能堆在一起。

---

## 14. MVP 建议与验收标准

### Phase 1

先做最小闭环：

- Workspace Home
- Explorer
- Document Viewer
- 基础标签筛选
- 文件预览
- 最近变更列表

Phase 1 验收标准：

- 能完成一次工作区全量扫描，并在 GUI 中展示结果
- 能按 metadata 字段进行基础筛选，例如领域、标签、状态、时间
- 能预览 Markdown 文档和常见资产类型
- 能从 GUI 跳转到 `Obsidian`、`VSCode` 或原始目录
- 所有功能均为只读，不产生任何文件写操作

### Phase 2

增强可视化能力：

- Knowledge Graph
- Timeline
- Collections
- Assets 页面
- 基础快照管理

Phase 2 验收标准：

- 能从索引结果中展示基础关系图和时间线
- `Collections` 可以以虚拟视图方式保存并复用聚合结果
- `Assets` 页面能稳定浏览常见非文本资料
- 基础快照能力有统一入口，但不要求完整版本治理体系

### Phase 3

补齐长期治理能力：

- 完整备份与迁移页面
- 完整性检查
- 路径修复工具
- 版本比较
- 多工作区支持

Phase 3 验收标准：

- 能展示完整的备份、迁移和一致性检查结果
- 能发现并提示路径异常、缺失 metadata 和孤立资源
- 能管理多工作区并保持统一的浏览体验
- 需要进一步定义 `Snapshots` 与历史版本的正式模型后再进入完整实现

---

## 15. 当前待定项

以下内容先明确为待定，不在本版文档强行细化：

- `Document`、`Asset`、`Collection`、`Snapshot` 等对象的数据契约
- `Snapshots` 的生成方式、保留策略和恢复模型
- 历史版本的来源，是来自 Git、快照系统还是独立版本存储

这些问题应在索引服务与备份机制进入实现前单独补一份规格。

---

## 16. 一句话定义

`Workspace GUI` 是 jarvis 的资料操作系统界面，它围绕真实文件工作区提供浏览、组织、检索、关系可视化、备份与迁移能力，确保所有知识与资产既可读、又可管、还能长期保留。
